## What upstream #386 does

A squash of several sub-commits that, net, (a) registers `claude-opus-4-8` in the
Anthropic / Bedrock / Vertex model registries, (b) adds the **full** Bedrock
adaptive-thinking machinery (Opus/Sonnet 4.7 + 4.8: `thinking.type:"adaptive"` +
`output_config.effort:"xhigh"`, temperature omitted in both `createMessage` and
`completePrompt`), (c) makes the direct Anthropic + Vertex providers emit adaptive
thinking for binary-reasoning models via a `getAnthropicProviderReasoning` helper,
(d) a comment-only touch to `openai.ts`, and (e) a Bedrock e2e smoke test.

## Fork-vs-upstream divergence analysis (why scope is trimmed)

The fork's provider layer is **behind** upstream on reasoning infrastructure that
#386's diff _assumes already exists in its parent_:

| Area                                    | Fork state                                                                                                             | Decision                                                                                                                                    |
| --------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `packages/types` bedrock.ts             | **== upstream parent**                                                                                                 | clean `git apply` of the bedrock-types hunk (adds opus-4-7 **and** opus-4-8 + 1M/global ID lists)                                           |
| `packages/types` anthropic.ts           | already has opus-4-7                                                                                                   | additive: insert opus-4-8 block                                                                                                             |
| `packages/types` vertex.ts              | already has opus-4-7                                                                                                   | additive: add `supportsReasoningBinary` to 4-7 + opus-4-8 block + 1M list entry                                                             |
| `src/api/providers/anthropic.ts`        | has `case "claude-opus-4-7"` in both caching switches                                                                  | additive: add `case "claude-opus-4-8"`                                                                                                      |
| `src/api/providers/bedrock.ts`          | **no** adaptive machinery at all (no `isGen47Model`); `RooCode#` userAgentAppId                                        | port the **whole** adaptive machinery (self-contained in #386); **preserve `RooCode#`**, neutralize the "Zoo Code UI" comment               |
| `src/api/providers/anthropic-vertex.ts` | uses `getModelParams`→`getAnthropicReasoning` (budget_tokens); **lacks** `getAnthropicProviderReasoning`               | **skip** the provider-logic change — depends on a helper from a _separate_ upstream PR; opus-4-8 inherits the fork's existing opus-4-7 path |
| `src/api/providers/openai.ts`           | **lacks** the `supportsTemperature !== false` guard (a prior 4.7-era PR); heavily diverged (GLM thinking, lazy client) | **skip** — #386's diff here is comment-only and refers to a guard the fork never took                                                       |
| `apps/vscode-e2e` bedrock e2e           | n/a                                                                                                                    | **skip** — not exercised by the build/vitest gate; harness-dependent, unverifiable in this round                                            |

**Key fact proven before deciding:** the fork's direct Anthropic + Vertex providers
do **not** emit `{type:"adaptive"}` for opus-4-7 today (no adaptive code outside
`base-openai-compatible-provider.ts`; `model-params.ts` uses `getAnthropicReasoning`
→ budget_tokens; the fork's `anthropic.spec.ts` opus-4-7 tests assert no adaptive
payload). So registering opus-4-8 with identical flags makes it behave **exactly**
like the fork's existing opus-4-7. Bringing in adaptive-everywhere would mean porting
a different PR's `getAnthropicProviderReasoning` refactor and changing the already-
working 4-7 behavior — out of scope for "port ONE PR".

Net effect of this port: **Opus 4.8 is selectable on all three providers; Bedrock
handles it fully correctly (adaptive thinking, no temperature); Anthropic-direct &
Vertex treat it identically to the fork's existing Opus 4.7.**
